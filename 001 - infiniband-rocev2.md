# GPU들이 대화하는 방법: InfiniBand vs RoCEv2 완전 정리

> InfiniBand와 RoCEv2 — 대규모 AI 훈련을 가능하게 만드는 두 가지 고속 네트워크 기술을 처음부터 파헤칩니다

---

## 01 — 들어가며: GPU가 아무리 빨라도, 통신이 느리면 소용없다

요즘 AI 뉴스를 보면 항상 GPU 개수 이야기가 나옵니다. "이번 모델은 GPT-4보다 두 배 많은 GPU를 사용했다", "슈퍼컴퓨터 클러스터에 GPU가 몇 만 개 달려있다"는 식입니다. 그런데 막상 이 GPU들이 어떻게 서로 협력하는지는 잘 다루지 않습니다.

현실을 보면, 대형 언어 모델 하나를 훈련하려면 **단일 GPU로는 불가능**합니다. GPT-3는 파라미터가 1,750억 개인데, A100 GPU 하나의 메모리는 80GB에 불과합니다. 모델 파라미터만 해도 약 350GB가 필요하니(FP16 기준), 물리적으로 들어가지 않습니다. 수십에서 수천 개의 GPU를 묶어 함께 사용할 수밖에 없습니다.

문제는 여기서 시작됩니다. GPU를 여러 개 묶으면, 이 GPU들은 매 훈련 스텝마다 서로 엄청난 양의 데이터를 주고받아야 합니다. **이 통신이 느려지는 순간, 수천 개의 GPU가 전부 놀게 됩니다.** 마치 100명의 요리사를 고용했는데 주방에 창구가 하나뿐인 것처럼요.

| 수치      | 설명                           |
| ------- | ---------------------------- |
| 1,750억  | GPT-3 파라미터 수 — 단일 GPU로 훈련 불가 |
| < 1μs   | InfiniBand 단방향 레이턴시 목표       |
| 400Gbps | 최신 스위치 포트당 대역폭               |

그래서 AI 인프라 엔지니어들이 공을 들이는 분야가 바로 **GPU 간 네트워크**입니다. 이 글에서는 이 문제를 해결하기 위해 실제로 사용되는 두 가지 기술, **InfiniBand**와 **RoCEv2**를 처음부터 차근차근 이해해 보겠습니다.

## 02 — 배경 지식: 분산 학습에서 GPU들은 무슨 이야기를 나눌까

네트워크 이야기를 본격적으로 하기 전에, GPU들이 왜 그리고 얼마나 자주 서로 통신해야 하는지를 먼저 이해하면 좋습니다.

### AllReduce — 핵심 집단 연산

가장 일반적인 분산 학습 방식인 **데이터 병렬화(Data Parallelism)** 에서는, 각 GPU가 서로 다른 배치(batch)의 데이터를 처리한 뒤 각자 계산한 **그래디언트(gradient)** 를 모아서 평균을 냅니다. 이 과정을 `AllReduce`라고 합니다.

문제는 이 그래디언트의 크기입니다. 파라미터가 700억 개인 모델(LLaMA-2 70B 정도)의 경우, 그래디언트 하나는 FP16(2바이트) 기준으로 약 **140GB**에 달합니다. 이게 훈련 스텝마다 모든 GPU 사이에서 교환되어야 합니다.

GPU 1,000개 클러스터에서 초당 1스텝을 돌린다면, 매초 140TB급 데이터가 네트워크를 타야 합니다. 이 수치가 얼마나 극단적인지는 가정용 기가비트 이더넷(1Gbps = 125MB/s)으로 계산해 보면 실감이 납니다. 답은 안 나옵니다.

### Bandwidth vs Latency — 무엇이 더 중요한가

네트워크 성능을 논할 때 두 가지 지표가 항상 등장합니다.

- **대역폭(Bandwidth):** 단위 시간당 전송할 수 있는 데이터 양. 큰 데이터를 옮기는 속도.
- **레이턴시(Latency):** 첫 비트가 목적지에 도착하기까지 걸리는 시간. 빠른 반응의 척도.

AllReduce처럼 대용량 그래디언트를 동기화하는 작업은 대역폭이 중요하지만, 수천 개 GPU가 서로 맞춰가며 진행하는 집단 연산에서는 **레이턴시 하나가 전체 파이프라인을 멈춰 세웁니다**. 이것이 바로 AI 훈련용 네트워크에서 마이크로초(μs) 단위 레이턴시를 집착하는 이유입니다.

## 03 — 문제 정의: 전통적인 방식 — CPU가 모든 것을 중재한다

일반적인 TCP/IP 네트워크에서 한 GPU가 다른 GPU에 데이터를 보내는 과정을 따라가 봅시다. 보기보다 훨씬 복잡합니다.

![그림 1: 전통적인 TCP/IP 6단계 데이터 경로](001%20-%20infiniband-rocev2.assets/traditional-path.svg)

GPU에서 데이터가 출발하면, 먼저 **시스템 메모리(RAM)로 복사**됩니다. CPU가 이를 받아 OS 네트워크 스택을 통해 NIC(네트워크 인터페이스 카드)로 전달합니다. NIC가 패킷을 보내면, 수신 측도 동일한 과정을 역순으로 밟습니다. 최종적으로 수신 GPU 메모리에 데이터가 도착하기까지 **총 6번의 메모리 복사와 2번의 OS 개입**이 발생합니다.

> **핵심 문제:** 이 방식에서는 CPU와 OS가 병목이 됩니다. 수천 개의 GPU가 동시에 통신을 요청하면, CPU는 인터럽트 처리와 메모리 복사로 포화 상태가 됩니다. GPU들은 CPU가 처리해줄 때까지 기다려야 합니다.

## 04 — 핵심 기술: RDMA — CPU를 우회해 메모리에 직접 접근한다

이 문제를 해결하기 위해 등장한 기술이 **RDMA(Remote Direct Memory Access, 원격 직접 메모리 접근)** 입니다. 이름 그대로, 원격 머신의 메모리에 CPU를 거치지 않고 직접 읽고 쓸 수 있는 메커니즘입니다.

### 두 가지 핵심 개념

**커널 바이패스(Kernel Bypass):** 일반 네트워크 통신은 모든 데이터가 OS 커널을 통과합니다. RDMA는 이 경로를 완전히 건너뜁니다. 애플리케이션(이 경우 GPU 드라이버)이 NIC와 직접 통신하기 때문에, OS 스케줄러의 대기열에 쌓이지 않습니다.

**제로-카피(Zero-Copy):** 전통적인 방식에서는 GPU 메모리 → 시스템 RAM → 커널 버퍼 → NIC 버퍼처럼 데이터가 여러 번 복사됩니다. RDMA에서는 출발지 메모리에서 목적지 메모리로 **단 한 번**의 이동으로 처리됩니다.

### GPUDirect RDMA — NVIDIA의 최적화

RDMA만으로도 CPU를 우회할 수 있지만, 데이터의 출발점이 GPU 메모리라면 시스템 RAM을 거치는 문제가 여전히 남습니다. 이를 해결하기 위해 NVIDIA가 개발한 것이 **GPUDirect RDMA**입니다.

GPUDirect RDMA는 NIC가 PCIe 버스를 통해 GPU 메모리에 직접 접근할 수 있게 합니다. 데이터가 GPU 메모리에서 출발해 시스템 RAM에 전혀 들르지 않고 NIC를 통해 네트워크로 나갑니다.

![그림 2: GPUDirect RDMA 직접 경로](001%20-%20infiniband-rocev2.assets/rdma-path.svg)

> **정리:** RDMA는 메커니즘(개념)이고, **InfiniBand와 RoCEv2는 이 메커니즘을 구현하는 두 가지 서로 다른 방식**입니다.

## 05 — InfiniBand: 처음부터 다시 만든 네트워크 패브릭

InfiniBand는 1990년대 후반 고성능 컴퓨팅(HPC) 커뮤니티에서 탄생했습니다. 당시 표준 이더넷의 한계를 느낀 연구자들이 "그냥 새로 만들자"고 해서 나온 결과물입니다. TCP/IP, 이더넷, OS 네트워크 스택 — 이 모든 것을 버리고, **전용 목적으로 설계된 새로운 네트워크 패브릭**을 만들었습니다.

2020년 NVIDIA가 InfiniBand의 주요 제조사인 **Mellanox를 69억 달러에 인수**하면서, InfiniBand는 사실상 NVIDIA의 AI 인프라 전략과 하나로 묶이게 되었습니다. 현재 대규모 AI 훈련 클러스터의 상당수는 InfiniBand 기반입니다.

### 핵심 구성 요소

**HCA(Host Channel Adapter, 호스트 채널 어댑터)** 는 서버에 꽂히는 카드로, 일반 이더넷의 NIC에 해당합니다. 하지만 단순한 NIC가 아니라, RDMA 처리 로직이 하드웨어로 내장되어 있습니다. CPU 없이도 자율적으로 메모리 전송을 처리할 수 있습니다.

**IB 스위치**는 이더넷 스위치와 비슷한 역할이지만, IB 전용 프로토콜을 처리합니다. 대규모 클러스터에서는 Fat-tree 토폴로지로 구성되어 어떤 두 노드 사이에도 동일한 대역폭을 보장합니다.

**서브넷 매니저(Subnet Manager)** 는 IB 네트워크의 두뇌입니다. 네트워크 토폴로지를 파악하고, 각 HCA에 주소를 할당하며, 라우팅 테이블을 관리합니다. 이더넷에서는 이 역할을 스위치들이 분산해서 담당하지만, IB에서는 중앙화된 컨트롤러가 처리합니다.

### 왜 빠른가: Lossless + Credit-Based Flow Control

InfiniBand의 성능 비밀은 **무손실(Lossless) 네트워크**에 있습니다. 일반 이더넷에서는 스위치 버퍼가 가득 차면 패킷을 버립니다(drop). 패킷을 잃으면 재전송이 필요하고, 이게 레이턴시를 폭발적으로 늘립니다.

InfiniBand는 **크레딧 기반 흐름 제어(Credit-Based Flow Control)** 를 사용해 이를 원천 차단합니다. 송신 측은 수신 측이 "지금 받을 수 있어"라는 크레딧을 줄 때만 패킷을 보낼 수 있습니다. 버퍼 오버플로가 구조적으로 불가능하기 때문에, 패킷 손실도 없고 재전송도 없습니다.

> InfiniBand의 레이턴시는 **1마이크로초(μs) 이하**입니다. 최신 NDR(Next Data Rate) 스펙은 포트당 400Gbps를 지원합니다. 비교하자면, 일반 데이터센터 이더넷의 레이턴시가 수십~수백 마이크로초 수준입니다.

### InfiniBand 장단점 요약

|        | 내용                                                              |
| ------ | --------------------------------------------------------------- |
| **강점** | 극도로 낮은 레이턴시(1μs 미만), 하드웨어 기반 RDMA, Lossless 구조, 대규모 확장에도 성능 일관성 |
| **약점** | 높은 하드웨어 비용, NVIDIA 벤더 종속, 전용 전문가 필요, 기존 IP 네트워크와 통합 불가          |

## 06 — RoCEv2: 기존 이더넷 위에 RDMA를 얹다

InfiniBand가 성능 면에서 탁월하지만, 모두가 처음부터 새로운 네트워크 패브릭을 깔 수 있는 건 아닙니다. 대부분의 데이터센터는 이미 이더넷 스위치와 케이블로 가득 차 있습니다. **기존 인프라를 최대한 활용하면서 RDMA의 이점을 누릴 수는 없을까?** 이 질문에서 RoCEv2가 탄생했습니다.

**RoCEv2(RDMA over Converged Ethernet version 2)** 는 RDMA 패킷을 `UDP/IP` 위에 실어 표준 이더넷 인프라로 전송합니다. 버전 1이 Layer 2(같은 이더넷 세그먼트)에서만 동작했던 것과 달리, v2는 Layer 3(IP 라우팅) 위에서 동작하기 때문에 여러 서브넷에 걸친 클러스터도 구성할 수 있습니다.

### 프로토콜 스택 비교

![그림 3: InfiniBand vs RoCEv2 프로토콜 스택 비교](001%20-%20infiniband-rocev2.assets/protocol-stacks.svg)

중요한 점은 **상위 API(IB Verbs)가 동일하다**는 것입니다. AI 프레임워크(NCCL 등)는 InfiniBand를 쓰든 RoCEv2를 쓰든 같은 인터페이스를 사용합니다. 차이는 그 아래 레이어에 있습니다.

### RoCEv2의 핵심 문제: 이더넷은 패킷을 버린다

RDMA가 제대로 동작하려면 **무손실(Lossless) 네트워크**가 전제되어야 합니다. 패킷이 중간에 사라지면 재전송이 필요하고, RDMA의 저레이턴시 이점이 사라집니다.

그런데 이더넷은 기본적으로 패킷을 버립니다. 스위치 큐가 가득 차면, 들어오는 패킷을 그냥 drop합니다. 이것이 RoCEv2의 핵심 과제이고, 이를 해결하기 위한 세 가지 메커니즘이 존재합니다.

### PFC — Priority Flow Control (우선순위 흐름 제어)

`PFC`는 스위치 큐가 일정 임계치를 넘으면, 특정 우선순위 클래스의 프레임을 일시적으로 **PAUSE** 시키는 메커니즘입니다. 상류 장비(upstream device)에게 "잠깐 멈춰"라고 신호를 보내는 것입니다.

PFC는 이더넷을 "ConvEth(Converged Ethernet)"으로 만드는 핵심 기술입니다. 하지만 완벽하지 않습니다. PAUSE 신호가 체인처럼 전파되어 연쇄적으로 트래픽을 막는 **HoL 블로킹(Head-of-Line Blocking)** 문제가 발생할 수 있습니다.

### ECN — Explicit Congestion Notification (명시적 혼잡 알림)

`ECN`은 혼잡이 감지될 때 패킷을 버리는 대신 **패킷에 마킹**을 하는 방식입니다. 스위치가 "나 지금 바빠"라는 신호를 패킷에 담아 보내면, 수신자가 이를 송신자에게 알립니다. 송신자는 이를 받아 전송 속도를 줄입니다.

패킷을 버리지 않기 때문에 재전송이 없고, 혼잡에 더 부드럽게 반응할 수 있습니다. 기존 TCP의 혼잡 제어(패킷 drop을 감지해 윈도우를 줄이는 방식)보다 빠르게 반응합니다.

### DCQCN — 두 메커니즘을 통합한 혼잡 제어

`DCQCN(Data Center Quantized Congestion Notification)`은 PFC와 ECN을 결합한 혼잡 제어 프로토콜입니다. ECN 신호를 받으면 송신 속도를 정량적으로 줄이고, 혼잡이 해소되면 다시 속도를 높이는 알고리즘입니다. Microsoft와 Mellanox가 공동 제안한 이 프로토콜은 현재 RoCEv2 환경의 사실상 표준입니다.

> **주의:** RoCEv2의 성능은 **PFC, ECN, DCQCN의 정교한 튜닝**에 크게 좌우됩니다. 이 파라미터들을 잘못 설정하면 HoL 블로킹이나 처리량 저하가 발생합니다. "이더넷이라 쉬울 것 같다"는 생각은 큰 오산입니다.

### RoCEv2 장단점 요약

|        | 내용                                                        |
| ------ | --------------------------------------------------------- |
| **강점** | 기존 이더넷 인프라 재활용, 상대적으로 낮은 비용, Layer 3 라우팅 지원, 다양한 벤더 선택 가능 |
| **약점** | PFC/ECN/DCQCN 정밀 튜닝 필수, 비결정론적 레이턴시, 대규모 확장 시 복잡도 증가       |

| -------- | ------------------------------ | -------------------------- |
| 네트워크 기반  | 전용 패브릭 (표준 이더넷과 분리)            | 표준 이더넷 (기존 인프라 재활용)        |
| RDMA 구현  | 하드웨어 내장 (HCA에서 처리)             | NIC + 소프트웨어 설정 필요          |
| 레이턴시     | **1μs 미만** (결정론적)              | 수 μs (튜닝에 따라 변동)           |
| 대역폭      | HDR 200Gbps / NDR 400Gbps      | 400G+ 이더넷 적용 가능            |
| Lossless | 구조적 보장 (Credit-based)          | PFC/ECN 설정으로 달성 (조건부)      |
| 혼잡 제어    | Credit-based flow control (HW) | DCQCN (SW + HW 조합)         |
| 라우팅      | IB Subnet 내 (Layer 2 수준)       | IP 라우팅 가능 (Layer 3)        |
| 벤더       | NVIDIA(Mellanox) 사실상 독점        | 다양한 벤더 (Broadcom, Intel 등) |
| 초기 비용    | 높음 (전용 HCA + 스위치)              | 상대적으로 낮음                   |
| 운용 복잡도   | 전문 지식 필요 (IB 특화)               | 이더넷 지식 + RDMA 튜닝 필요        |
| 주요 사용처   | HPC 슈퍼컴, NVIDIA DGX 클러스터       | AWS, Azure, GCP 등 클라우드     |

## 08 — 선택 가이드: 어떤 상황에서 무엇을 선택할까?

사실 이 질문은 "어떤 기술이 더 뛰어난가?"가 아닙니다. 두 기술 모두 실제로 수백, 수천 개의 GPU 클러스터에서 잘 동작하고 있습니다. 진짜 질문은 **"우리 상황에 무엇이 맞는가?"** 입니다.

### InfiniBand를 선택할 때

- 절대적인 최고 성능이 우선순위인 경우
- 전용 AI 훈련 클러스터를 새로 구축하는 경우
- 예측 가능한 레이턴시가 반드시 필요한 워크로드
- NVIDIA GPU + DGX 시스템 기반의 환경
- 비용보다 성능을 중시하는 연구/훈련 환경

### RoCEv2를 선택할 때

- 기존 이더넷 인프라를 최대한 활용하고 싶을 때
- 클라우드 기반 분산 학습 환경 (AWS, Azure 등)
- 멀티-벤더 환경에서 유연성이 필요한 경우
- 비용 효율성이 중요한 대규모 추론(inference) 클러스터
- 이더넷 기반의 기존 데이터센터 팀이 운용하는 환경

> 현실적으로, 대형 클라우드 제공업체들은 대부분 RoCEv2 기반으로 자체 AI 클러스터를 운영합니다. 이미 이더넷 인프라를 대규모로 보유하고 있고, 전담 네트워크 엔지니어링 팀이 있기 때문입니다. 반면 연구소나 AI 기업이 전용 훈련 클러스터를 직접 구축할 때는 InfiniBand를 선택하는 경우가 많습니다.

## 09 — 마치며

이 주제를 처음 접했을 때는 "GPU 간 네트워크쯤이야 그냥 빠른 이더넷 쓰면 되는 거 아닌가?"라고 생각했습니다. 그런데 파고들다 보니, AI 훈련의 성능 최적화에서 네트워크가 차지하는 비중이 상상 이상으로 크다는 걸 알게 되었습니다.

InfiniBand와 RoCEv2는 서로 다른 철학을 가지고 있습니다. InfiniBand는 "기존 것을 버리고 완벽한 것을 만들자"는 접근이고, RoCEv2는 "기존 것 위에서 최대한 잘 해보자"는 접근입니다. 둘 다 장단점이 명확하고, 상황에 따라 정답이 다릅니다.

앞으로의 방향을 보면, 이더넷 진영도 손 놓고 있지 않습니다. **Ultra Ethernet Consortium** 이라는 업계 컨소시엄이 AI 워크로드에 최적화된 새로운 이더넷 표준을 개발 중입니다. 400G, 800G 시대가 도래하면서 두 기술의 경쟁은 더욱 흥미로워질 것 같습니다.

네트워크 공부를 하다 보면 OSI 7계층, TCP/IP 스택처럼 익숙한 개념들이 AI 인프라 세계에서 얼마나 다르게 적용되는지 새삼 느끼게 됩니다. 다음에는 GPU 클러스터의 토폴로지(Fat-tree, Rail-Optimized 등)나 NCCL 내부 동작 방식으로 이어가 볼 생각입니다.
